# Дизайн: боевая защита соединения

Дата: 2026-05-13

## Цель

Сделать настоящий проверяющий слой для боевого соединения клиента с серверами.
Клиент должен доверять серверу только тогда, когда одновременно выполнены два
условия:

- обычная проверка сертификата прошла успешно;
- публичный ключ сертификата совпал с заранее закреплённым ключом для этого
  имени сервера.

Эта итерация не открывает публичный запуск клиента через внешний интерфейс.
Он остаётся закрытым, пока не готовы настоящая проверка устройства, мобильные
мосты и серверная связка.

## Текущая причина

После Фазы 2 боевой транспорт честно отказывает: настройка проверяется, но
клиент не собирается, потому что проверка закреплённого ключа ещё не связана
с проверкой сертификата.

Это безопаснее, чем заглушка, но ещё не боевой путь. Следующий шаг — заменить
честный отказ настоящей проверкой закреплённого ключа.

## Выбранный подход

Выбран подход с усилением обычной проверки сертификата.

Сначала выполняется системная проверка сертификата: срок действия, имя сервера
и цепочка доверия. Если она отказывает, соединение отказывается.

Только после этого выполняется проверка Umbrella: из публичной части
сертификата считается 32-байтовый хеш, и он сравнивается с закреплёнными
ключами для имени сервера.

Так мы не заменяем стандартную проверку, а добавляем к ней вторую границу.

## Архитектура

В `umbrella-client` появляется отдельный проверяющий слой для закреплённых
ключей.

Он получает:

- имя сервера;
- сертификат сервера;
- карту `имя сервера -> разрешённые закреплённые ключи`;
- внутренний проверяющий сертификата.

Порядок работы:

1. Внутренний проверяющий сертификата проверяет обычные правила доверия.
2. Если обычная проверка прошла, слой Umbrella извлекает публичную часть
   сертификата.
3. Из этой публичной части считается закреплённый ключ.
4. Ключ сравнивается с основным и запасным ключом для имени сервера.
5. Совпадение разрешает соединение.
6. Несовпадение или неизвестное имя сервера запрещает соединение.

Боевой сборщик транспорта начинает собирать клиент только при полной боевой
настройке:

- ровно пять запечатанных серверов;
- почтовый сервер;
- сервер журнала ключей;
- релей звонков;
- везде `https`;
- нет тестовых адресов;
- у каждого имени сервера есть закреплённый ключ.

## Ошибки

Ошибки должны быть явными и проверяемыми:

- обычная проверка сертификата отказала;
- для имени сервера нет закреплённого ключа;
- закреплённый ключ не совпал;
- сертификат пустой или не разбирается;
- настройка похожа на тестовую;
- адрес не `https`.

Любая такая ошибка закрывает соединение до отправки запроса.

## Реальные атачные тесты

Тесты должны проверять не текст реализации, а защиту от обходов:

1. Правильный сертификат и правильный закреплённый ключ проходят.
2. Тот же сервер с другим ключом отклоняется.
3. Запасной закреплённый ключ проходит, чтобы поддержать смену сертификата.
4. Неизвестное имя сервера отклоняется.
5. Тестовые адреса, `http`, `localhost`, `127.0.0.1` и `.invalid`
   отклоняются на проверке настройки.
6. Пустой или неверно закодированный сертификат возвращает ошибку, а не
   падение процесса.
7. Опасные режимы принятия любого сертификата или любого имени сервера не
   используются.
8. Внутренний боевой сборщик транспорта с полной настройкой перестаёт
   возвращать ошибку “проверяющий не подключён”.
9. Публичный внешний запуск клиента всё ещё возвращает отказ, потому что
   остальные ворота не закрыты.

## Что меняем

- `crates/umbrella-client/src/transport/pinning.rs` — настоящая проверка
  закреплённого ключа.
- `crates/umbrella-client/src/transport/http2_client.rs` — подключение
  проверяющего слоя к боевому сборщику транспорта.
- `crates/umbrella-client/Cargo.toml` и корневой `Cargo.toml` — только если
  нужна прямая зависимость для проверки сертификата.
- `crates/umbrella-ffi/tests/production_bootstrap.rs` — проверка, что
  публичный внешний запуск всё ещё закрыт по оставшимся воротам.
- `docs/security/production-readiness-boundaries.md` — обновление текущей
  границы боевой готовности.

## Что не меняем

В этой итерации не делаем:

- настоящую проверку iOS, Android и Web-устройства;
- мобильные мосты;
- серверное развёртывание;
- сверку свидетелей журнала ключей;
- публичное открытие боевого клиента.

## Проверки перед завершением

Минимальные ворота:

- точечные тесты транспорта;
- тесты публичного внешнего запуска;
- `cargo fmt --all -- --check`;
- `cargo clippy --workspace --all-targets --all-features --locked -- -D warnings`;
- `cargo test --workspace --all-features --locked`.

Если подключение готового TLS-стека через клиентский сборщик окажется
нестабильным из-за версий, реализация должна остановиться честно: проверяющий
модуль и атачные тесты остаются, а боевой сборщик продолжает закрываться
понятной ошибкой. Ложный боевой путь запрещён.
