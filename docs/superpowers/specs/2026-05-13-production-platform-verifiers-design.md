# Дизайн: локальная основа платформенных проверяющих

Дата: 2026-05-13

## Цель

Сделать следующий слой боевой защиты для проверки устройства: локальную основу
настоящих платформенных проверяющих для Айос, Андроида и веба.

Цель этой фазы не в том, чтобы открыть публичный запуск клиента. Цель в том,
чтобы убрать последнюю опасную серую зону: код не должен принимать настоящий
вид платформы только потому, что токен непустой или похож на правильный.

После этой фазы боевой серверный путь должен уметь:

- проверять общие обязательные поля платформенного доказательства;
- привязывать доказательство к серверному вызову;
- привязывать доказательство к приложению или сайту;
- привязывать доказательство к ключу устройства;
- проверять время и счётчики там, где платформа их даёт;
- закрыто отказывать, если для полной проверки нужны внешние ключи, корни
  доверия, учётные записи или живая связка, которых нет в локальном
  репозитории.

## Текущая правда

В предыдущей фазе уже сделан строгий серверный каркас для `umbrella-backup` и
`umbrella-oprf`. Он проверяет подпись, серверный вызов, свежесть, состояние
устройства, платформу и явный платформенный проверяющий.

Сейчас `Platform::Testing` отвергается в боевом пути. Айос, Андроид и веб пока
тоже закрыто отказывают через неподключённый проверяющий. Это безопасно, но
ещё не даёт локального слоя, который умеет отличить правильно оформленное
платформенное доказательство от мусора, подмены или чужого приложения.

Публичный запуск клиента через внешний интерфейс остаётся закрытым. Его нельзя
открывать, пока не связаны настоящие платформенные проверяющие, мобильные
мосты и серверная интеграция.

## Выбранный подход

Выбран строгий локальный каркас настоящих платформенных проверяющих.

Мы не делаем фальшивый боевой успех. Если для полной проверки нужен живой
сервис Эпл или Гугл, закрытый ключ из панели разработчика, корневой сертификат,
учётная запись или мобильный мост, локальный код должен вернуть понятный
отказ.

При этом всё, что можно проверить локально и честно, должно проверяться уже в
этой фазе. Это даёт пользу без самообмана:

- будущие настоящие проверки подключатся в готовые строгие места;
- тесты будут ловить подмену вызова, приложения, сайта, ключа и счётчика;
- публичные заявления останутся честными;
- серверный путь останется закрытым до полной связки.

## Архитектура

Появляется отдельный слой платформенной проверки. Его задача — не выдавать
долю ключа или ответ поиска контактов, а только сказать: можно ли доверять
платформенному доказательству для этого устройства и этого серверного вызова.

Слой должен быть маленьким и разделённым по платформам.

### 1. Общий вход проверки

Общий вход содержит:

- платформу;
- байты токена;
- серверный вызов;
- публичный ключ устройства;
- имя приложения или сайта;
- текущее серверное время;
- сохранённое состояние платформенной записи, если оно уже было заведено;
- строгие настройки боевого окружения.

Общий вход не должен содержать тестового режима. Тесты могут использовать
отдельные тестовые типы, но боевой путь не должен принимать их как настоящие.

### 2. Айос

Для Айос локальный слой делится на две части.

Первая часть — первичная привязка ключа приложения. Она должна проверять
структуру доказательства, связь с серверным вызовом, имя приложения, среду,
ключ устройства и начальный счётчик.

Вторая часть — повторное доказательство при запросе. Она должна проверять
подпись, серверный вызов, имя приложения, сохранённый ключ и рост счётчика.

Если в локальном репозитории нет нужного корня доверия или полного разбора
формата, проверка должна вернуть отказ "проверяющий не полностью подключён".
Нельзя принимать токен Айос по форме без проверки цепочки доверия и привязки к
приложению.

### 3. Андроид

Для Андроида токен является вложенным подписанным и зашифрованным сообщением.
Локальный слой должен проверять:

- что токен имеет допустимую форму;
- что серверный вызов внутри ответа совпадает с выданным сервером вызовом;
- что имя приложения совпадает с ожидаемым;
- что ответ не старый;
- что итог устройства и приложения разрешён политикой.

Если нет ключей из панели Гугл, доступа к серверной проверке Гугл или полной
локальной расшифровки и проверки подписи, путь должен закрыто отказать. Нельзя
принимать Андроид только по распарсенному тексту.

### 4. Веб

Для веба локальный слой должен проверять утверждение входа:

- вызов совпадает с серверным вызовом;
- сайт совпадает с ожидаемым сайтом;
- хеш домена в данных устройства совпадает;
- флаги присутствия и проверки пользователя соответствуют политике;
- подпись проходит на сохранённом публичном ключе;
- счётчик не идёт назад.

Если сохранённого ключа нет, сайт не совпал, подпись не прошла или счётчик
подозрительный, путь закрыто отказывает.

### 5. Связка с backup и OPRF

`umbrella-backup` и `umbrella-oprf` уже имеют интерфейс боевого платформенного
проверяющего. Новый слой должен подключаться через этот интерфейс, не ломая
порядок проверки:

1. подпись запроса;
2. серверный вызов;
3. свежесть;
4. состояние устройства;
5. платформенная проверка.

Если ранний шаг отказал, платформенный проверяющий не вызывается. Это уже
покрыто предыдущей фазой и должно остаться неизменным.

## Ошибки

Ошибки должны быть точными и проверяемыми:

- неподдержанная платформа;
- тестовая платформа в боевом пути;
- токен пустой или слишком большой;
- неверная форма платформенного токена;
- серверный вызов не совпал;
- приложение или сайт не совпали;
- ключ устройства не совпал;
- подпись платформенного доказательства не прошла;
- счётчик не вырос;
- доказательство слишком старое;
- нет обязательных боевых настроек;
- внешний корень доверия или ключ проверки не подключён.

Для внешнего пользователя сообщение может быть обобщённым, но тесты должны
видеть точную причину.

## Реальные атакующие тесты

Тесты должны доказывать защиту, а не совпадение с удобной заглушкой.

Нужны проверки:

1. Тестовая платформа отвергается даже с правильной подписью запроса.
2. Айос с чужим серверным вызовом отвергается.
3. Айос с чужим приложением отвергается.
4. Айос с неверным ключом устройства отвергается.
5. Айос со старым или нерастущим счётчиком отвергается.
6. Андроид с чужим серверным вызовом отвергается.
7. Андроид с чужим именем приложения отвергается.
8. Андроид без ключей проверки закрыто отказывает.
9. Веб с чужим сайтом отвергается.
10. Веб с чужим вызовом отвергается.
11. Веб с неверной подписью отвергается.
12. Веб со счётчиком назад отвергается.
13. Любая платформа с пустым токеном отвергается.
14. Любая платформа с чрезмерно большим токеном отвергается.
15. `umbrella-backup` и `umbrella-oprf` используют одинаковую политику отказов.

## Что меняем

План внедрения должен затронуть:

- новый модуль или малый общий крейт для платформенных проверяющих;
- `crates/umbrella-backup/src/cloud_wrap/signed_request.rs`;
- `crates/umbrella-oprf/src/attestation.rs`;
- ошибки `umbrella-backup` и `umbrella-oprf`;
- тесты для Айос, Андроида и веба;
- тесты связки с `umbrella-backup` и `umbrella-oprf`;
- `docs/security/production-readiness-boundaries.md`;
- `README.md` и `docs/README.md`, если изменится публичный статус.

Если общий слой начнёт сильно дублироваться между backup и OPRF, нужно вынести
его в отдельный малый Rust-крейт внутри workspace. Если дублирование останется
тонким и ясным, можно оставить его рядом с текущими модулями.

## Что не меняем

В этой фазе не открываем публичный запуск клиента.

В этой фазе не подделываем живые сервисы Эпл и Гугл.

В этой фазе не считаем мобильные мосты боевыми, если они только передают байты
и не связаны с настоящей платформенной проверкой.

В этой фазе не принимаем токен, если не можем проверить его подпись, корень
доверия, вызов и привязку к приложению или сайту.

## Проверки перед завершением

Минимальные ворота:

```bash
cargo test -p umbrella-backup --all-features --locked
cargo test -p umbrella-oprf --all-features --locked
cargo test -p umbrella-client --all-features --locked
cargo test -p umbrella-ffi public_bootstrap --locked
cargo test -p umbrella-ffi public_bootstrap --features pq --locked
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --workspace --all-features --locked
cargo test --workspace --all-features --locked
```

Если появятся новые зависимости для разбора токенов, нужно отдельно прогнать
проверку зависимостей и обновить документ политики зависимостей.

## Свежие источники

Проверено 2026-05-13:

- Apple App Attest:
  https://developer.apple.com/documentation/devicecheck/validating-apps-that-connect-to-your-server
- Google Play Integrity:
  https://developer.android.com/google/play/integrity/classic?hl=en
- WebAuthn Level 3:
  https://www.w3.org/TR/webauthn-3/

Из этих источников для текущей фазы важны общие правила: сервер выдаёт
одноразовый вызов, платформа должна включить этот вызов в доказательство, а
сервер должен проверить подпись, привязку к приложению или сайту и состояние
устройства. Простая проверка "токен непустой" не является защитой.

## Готовность фазы

Фаза готова, когда:

- есть явный локальный слой платформенных проверяющих;
- Айос, Андроид и веб не проходят по одному только внешнему виду токена;
- каждый неподключённый внешний ключ или корень доверия закрывает путь отказом;
- `umbrella-backup` и `umbrella-oprf` вызывают платформенный слой только после
  подписи, свежести и состояния устройства;
- тесты показывают реальные отказы на подмену вызова, приложения, сайта,
  ключа, подписи и счётчика;
- документы не обещают открытый публичный запуск;
- рабочая папка чистая после записи в историю.
