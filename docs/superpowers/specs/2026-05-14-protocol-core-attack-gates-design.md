# Дизайн: боевые атакующие ворота ядра протокола

Дата: 2026-05-14

## Цель

Фаза 3А усиливает само ядро Umbrella Protocol. Цель — не открыть публичный
боевой клиент, а сделать так, чтобы ключевые крейты имели реальные атакующие
проверки и честные границы:

- тестовый путь не выглядит боевым;
- заявленная защита либо реально проверяется, либо закрыто отказывает;
- атаки подмены, повтора, отката, неверного устройства, плохого закрепления
  сертификата и тихого downgrade имеют тесты;
- документы простым языком говорят, что закрыто, что ещё не боевое, и какие
  команды являются воротами выпуска.

Эта фаза не обещает невозможность взлома. Она уменьшает известные классы риска
и делает ложную готовность невозможной в коде, тестах и документах.

доступную часть, либо закрыто отказывать.

## Выбранный подход

Выбран подход "атакующая матрица + реальные тесты + закрытые границы".

Каждая итерация идёт так:

1. найти текущую границу и доказать риск или пробел;
2. написать красный атакующий тест или точную проверочную команду;
3. исправить первопричину, а не симптом;
4. обновить документы;
5. выполнить фокусные проверки;
6. сделать отдельный коммит.

Если тест уже есть и честно покрывает атаку, задача этой фазы — не дублировать
его, а записать это в матрицу покрытия. Если тест звучит сильнее, чем реально
доказывает, его надо переименовать или разделить на честные тесты.

## Архитектурная граница

Фаза усиливает локальные защитные контракты между крейтами.

`umbrella-platform-verifier` отвечает за локально проверяемую форму
платформенного доказательства: размер, приложение или сайт, серверный вызов,
ключ, подпись, счётчик и запрет тестовой платформы в боевом пути.

`umbrella-client` отвечает за транспортные настройки, закрепление
SPKI-ключей, запрет тестовых адресов в боевой настройке и отсутствие тихого
перехода на слабый режим.

`umbrella-kt` отвечает за прозрачность ключей: корень без достаточных подписей
не должен приниматься как боевой, подмена корня должна отклоняться, а split-view
с большинством злых свидетелей должен быть описан честно как случай, который
требует самопроверки и публичной сверки наблюдений.

`umbrella-oprf` отвечает за приватный поиск контактов: повтор серверного вызова,
неверная платформа, неверные доли, повтор долей и подмена ответа должны иметь
проверяемое поведение.

`umbrella-backup` отвечает за развёртку резервного ключа: старый серверный вызов,
неизвестное устройство, неверная доля, откат версии и смешение тестового пути с
боевым должны закрываться.

`umbrella-sealed-sender` отвечает за скрытие отправителя: подмена envelope,
повреждение подписи, неверный получатель и повтор между версиями должны
отклоняться или менять результат так, чтобы оригинальная подмена не проходила.

| Область | Атака | Ожидаемое поведение |
|---|---|---|
| Устройство | `Platform::Testing` в боевом пути | отказ |

| Транспорт | конфликтующие pins для одного хоста | отказ настройки |

## Ошибки и отказы

Ошибки должны быть точными. Нельзя возвращать общий успех или общий сетевой сбой
там, где понятно, что причина — тестовая платформа, плохой pin, старый
серверный вызов, неизвестное устройство или неподключённый боевой проверяющий.

Боевой путь обязан закрываться раньше, чем начнёт использовать тестовый
конструктор, тестовый адрес, тестовую платформу или заглушку.

Если часть проверки зависит от настоящего сервера, внешнего доверенного корня
или мобильной связки, локальный код должен честно:

- проверить доступную часть;
- закрыто отказать в полном боевом режиме;
- записать оставшуюся границу в документы.

## Тестирование

Тесты должны быть атакующими, а не подогнанными под ответ. Для каждого
исправления нужен один из вариантов:

- новый тест сначала падает на текущем коде;
- существующий тест уже доказывает атаку, и это явно записано в матрице;
- команда проверки уже является воротами, и её результат записан в документе.

Минимальные фокусные команды для плана внедрения:

```bash
cargo test -p umbrella-platform-verifier --all-features --locked
cargo test -p umbrella-client --all-features --locked
cargo test -p umbrella-kt --all-features --locked
cargo test -p umbrella-oprf --all-features --locked
cargo test -p umbrella-backup --all-features --locked
cargo test -p umbrella-sealed-sender --all-features --locked
bash scripts/audit-public-access-notices.sh
```

Финальные ворота фазы:

```bash
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --workspace --all-features --locked
cargo test --workspace --all-features --locked
bash scripts/audit-public-access-notices.sh
```

Если полные ворота падают, применяется системный разбор: прочитать ошибку,
воспроизвести, найти первопричину, сравнить с рабочими примерами, затем чинить
одну причину за раз.

## Критерии готовности

Фаза 3А готова, когда:

- для каждой области из атакующей матрицы есть тест, явная граница или
  документированный отказ;
- найденные тестовые пути не выглядят боевыми;
- публичный FFI-запуск остаётся закрыт, если полная связка всё ещё не готова;
- документы и код не противоречат друг другу;
- фокусные проверки прошли;
- финальные ворота либо прошли полностью, либо падение разобрано и честно
  записано как блокер выпуска;
- каждая итерация имеет отдельный коммит.

## Риски

Главный риск — попытаться закрыть все классы атак в одной огромной правке. Это
сломает проверяемость. Поэтому план внедрения должен идти малыми срезами.

Второй риск — назвать локальный тест полной боевой защитой. Так делать нельзя:
локальный тест доказывает только локальную часть.

Третий риск — смешать эту фазу с серверной интеграцией. Сервер будет отдельной
фазой, когда появится готовая точка встраивания. Сейчас усиливается только
ядро Umbrella Protocol.
