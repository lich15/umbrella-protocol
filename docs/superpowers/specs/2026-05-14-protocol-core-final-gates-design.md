# Дизайн: финальные боевые ворота ядра протокола

Дата: 2026-05-14

## Цель

Цель этой фазы — довести само ядро Umbrella Protocol до честной боевой
границы перед будущим встраиванием в сервер. Мы не открываем публичный FFI
клиент, не подключаем настоящий сервер и не делаем полную проверку реальных
Apple/Android устройств. Эти части остаются последней фазой.

Внутри текущего репозитория нужно закрыть всё, что можно доказать локально:
тестовые пути, транспорт, закрепление ключей сертификата, KT, OPRF, резервное
копирование, скрытие отправителя, запрет тихого отката на слабый режим,
зависимости, проверки и документы.

В эту фазу не входит:

- папка `/Users/daniel/Documents/Projects/Messenger/rust_1mlrd`;

- обещание “невозможно взломать”.

Если локальный код не может доказать боевое свойство без сервера, внешнего
корня доверия или настоящего устройства, путь должен закрыто отказать и
документ должен назвать это границей выпуска.

## Выбранный подход

Выбран подход “боевые ворота ядра”.

Каждая итерация идёт одинаково:

1. прочитать код и документы по одной области;
2. найти реальную точку обхода или подтвердить, что тест уже честный;
3. если есть пробел, сначала написать атакующий тест;
4. исправить первопричину;
5. обновить документы простым русским языком;
6. прогнать фокусные проверки;
7. сделать отдельный коммит.

Тест не считается настоящим, если он просто повторяет ожидаемый ответ. Он
должен ломать путь атакой: подмена, повтор, откат, неверный ключ, неверная
версия, плохой pin, тестовая платформа или тихий downgrade.

## Области усиления

### 1. Тестовый путь не выглядит боевым

Все места с `for_test`, `Testing`, `TestOnly`, `mock`, `stub`, `fake` и
`placeholder` должны быть разделены на две группы:

- допустимо: код находится в тестах, под `cfg(test)` или явно назван как стенд;
- недопустимо: боевой конструктор может молча попасть в такой путь.

Публичный FFI должен продолжать закрыто отказывать, пока нет полной связки
транспорта, платформенных проверок, мобильного слоя и сервера.

### 2. TLS и закрепление SPKI-ключей

Транспорт должен доказывать три вещи:

- обычная проверка сертификата идёт первой;
- совпавший pin не может обойти плохую цепочку сертификата;
- плохой pin, неизвестный хост, локальный адрес, служебный адрес и слабая
  схема соединения отвергаются.

Если найдётся место, где создаётся HTTP/2 клиент без боевой проверки, оно
должно быть явно тестовым или закрыто отказать в боевом режиме.

### 3. KT

KT должен честно разделять два случая:

- подмена корня, эпохи, размера журнала или подписи отвергается локально;
- split-view при захвате достаточного числа свидетелей локально может пройти,
  поэтому нужен внешний обмен наблюдениями, самопроверка и операционная
  политика свидетелей.

Документы не должны говорить, что split-view полностью закрыт одним локальным
тестом.

### 4. OPRF

OPRF должен закрывать:

- повтор серверного вызова;
- неверную платформу;
- неверное или отозванное устройство;
- подмену blinded-запроса, токена, server nonce и device key;
- повтор или подмену доли так, чтобы результат не принимался как честный
  успех.

Тестовые приватные ключи и тестовое разбиение секрета должны оставаться
только тестовой областью.

### 5. Backup

Резервное копирование должно закрывать:

- повтор серверного вызова;
- неверное или отозванное устройство;
- подмену получателя, chat id, timestamp, токена, server nonce и device key;
- неверный AAD;
- смешение V1/V2 форматов;
- тихий откат на старую версию.

Если mock-транспорт используется в тесте, документ должен прямо говорить, что
это не боевая серверная готовность.

### 6. Sealed Sender

Скрытие отправителя должно отдельно доказывать:

- подмена ciphertext не проходит;
- неверный получатель не может открыть envelope;
- повтор сообщения к другому получателю не проходит;
- V1 не принимается как V2, V2 не принимается как V1;
- подмена подписи или внутренней личности отправителя не принимается;
- случайный мусор не вызывает panic и не принимается молча.

Если уже существующий тест это доказывает, матрица должна назвать его точно.

### 7. Зависимости и запреты

Ворота выпуска должны включать:

- проверку обычного дерева зависимостей;
- проверку правил `cargo-deny`, если инструмент доступен;
- запрет опасных заявлений в документах;
- аудит тестовых путей, чтобы они не выглядели боевыми;
- форматирование, анализатор Rust, документацию и полный прогон тестов.

Если внешний инструмент не установлен, это не маскируется как успех: причина
записывается как блокер или как явно ограниченная проверка.

## Проверки успеха

Фаза считается закрытой только если:

- для каждой области есть атакующий Rust-тест, честная граница или закрытый
  отказ;
- документы и код не противоречат друг другу;
- публичный FFI-запуск всё ещё закрыт;
- тестовые пути не выглядят боевыми;
- полные проверки Rust прошли или падение разобрано и записано;
- каждая законченная итерация имеет отдельный коммит.

Минимальные финальные команды:

```bash
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --workspace --all-features --locked
cargo test --workspace --all-features --locked
bash scripts/audit-protocol-core-attack-gates.sh
bash scripts/audit-public-access-notices.sh
bash scripts/audit-dependency-policy.sh
```

Дополнительные проверки добавляются в план реализации, если чтение кода покажет
конкретный риск.

## Риски

Первый риск — заявить больше, чем код доказывает. Противоядие: каждое
утверждение в документах должно иметь тест, закрытый отказ или честную границу.

Второй риск — смешать ядро протокола с будущим сервером. Противоядие: сервер
и реальные устройства остаются вне этой фазы.

Третий риск — сделать одну огромную правку. Противоядие: маленькие итерации,
фокусные проверки и коммит после каждой законченной части.
