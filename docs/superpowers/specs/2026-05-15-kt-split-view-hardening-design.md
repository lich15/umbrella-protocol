# Усиление KT против раздвоения журнала

Дата: 2026-05-15

Статус: дизайн утверждён устно, записан перед планом внедрения.

## Цель

Закрыть дыру в журнале ключей Umbrella Protocol: если три настроенных свидетеля
подписали две разные версии одной эпохи, один локальный клиент может принять
свою версию как валидную. Это не ошибка подписи Ed25519 и не ошибка дерева
Merkle. Это нехватка слоя, который сравнивает увиденные корни между клиентами,
свидетелями и наблюдателями.

Цель исправления: сделать раздвоение журнала доказуемым, сделать боевой путь
строго закрытым при конфликте, а публичные обещания привести к правде. Без
живой сети, настоящих свидетелей и обмена наблюдениями нельзя честно доказать,
что раздвоения никогда не было. Локально можно реализовать ядро проверки,
доказательство атаки, отказ при конфликте и атакующие тесты.

## Источники решений

- RFC 9162 Certificate Transparency: подписанные головы дерева, доказательства
  включения и неизменности: https://www.rfc-editor.org/rfc/rfc9162.html
- CONIKS: пользователи следят за своими ключами, а расхождение корней становится
  доказательством атаки: https://eprint.iacr.org/2014/1004.pdf
- IETF Key Transparency draft: клиенты видят много корней во времени и обязаны
  проверять их согласованность:
  https://www.ietf.org/archive/id/draft-ietf-keytrans-protocol-02.html
- Trillian: клиент хранит увиденные головы дерева и проверяет, что новая версия
  продолжает старую: https://transparency.dev/verifiable-data-structures/
- WhatsApp AKD и аудит Cloudflare: эпоха связывает номер, предыдущий корень и
  новый корень; внешний аудитор проверяет уникальность и правильность перехода:
  https://engineering.fb.com/2023/04/13/security/whatsapp-key-transparency/
  https://noise.getoto.net/2024/09/24/cloudflare-helps-verify-the-security-of-end-to-end-encrypted-messages-by-auditing-key-transparency-for-whatsapp/
- Consistency-or-Die: если клиент не может доказать согласованное состояние, он
  должен остановиться, а не продолжать как будто всё безопасно:
  https://eprint.iacr.org/2024/879

## Граница приватности

Публично нельзя хранить телефоны, имена, контакты, чаты, список устройств
пользователя, открытые ключи аккаунта в сыром виде или кто кого проверял.

Публичная запись наблюдения должна содержать только технические доказательства:

- идентификатор журнала без личных данных;
- номер эпохи;
- предыдущий корень;
- текущий корень;
- размер журнала;
- время подписи;
- подписи свидетелей;
- версию формата.

Локальный клиент может хранить малую историю увиденных корней на своём
устройстве. Эта история не уходит в общий журнал с именем пользователя. Если
нужно передать доказательство атаки, передаются две конфликтующие головы дерева
и подписи свидетелей, но не личная запись аккаунта.

## Причина дыры

Сейчас `verify_signed_epoch` проверяет одну подписанную голову эпохи:

- подписи от известных свидетелей;
- порог подписей;
- совпадение подписи с `epoch`, `root`, `log_size`, `timestamp`.

Этого достаточно против подмены одной головы, но недостаточно против двух
валидно подписанных голов одной эпохи. Функция не знает, что другому клиенту
показали другой корень. Текущий тест `split_view_exchange.rs` честно показывает
эту границу: две версии проходят локально, а простая сверка клиентов видит
расхождение.

Корневая причина: нет библиотечного слоя наблюдений, нет объекта доказательства
раздвоения, нет обязательной проверки цепочки эпох и нет состояния свидетеля,
которое запрещает подписывать второй другой корень для той же эпохи.

## Дизайн защиты

### 1. Наблюдение KT

Добавить в `umbrella-kt` отдельный тип наблюдения эпохи. Он описывает не
пользователя, а только увиденную голову журнала:

- версия формата;
- идентификатор журнала;
- эпоха;
- предыдущий корень;
- текущий корень;
- размер журнала;
- время;
- подписи свидетелей.

Такой объект можно безопасно сравнивать и публиковать, потому что он не содержит
личных данных. Тестовое поле вроде `client = "alice"` не должно стать боевым
полем.

### 2. Доказательство раздвоения

Добавить тип доказательства атаки. Он состоит из двух наблюдений, которые:

- относятся к одному журналу;
- относятся к одной эпохе;
- имеют разные корни или разные размеры журнала;
- обе проходят проверку порога подписей.

Если такое доказательство создано, результат проверки должен быть не
предупреждением, а критическим отказом. Клиент должен считать состояние KT
скомпрометированным до ручного разбора и смены доверенной конфигурации.

### 3. Цепочка эпох

Каждая новая эпоха должна быть связана с предыдущей. Проверка должна отклонять:

- откат на старую эпоху;
- повтор старой головы вместо новой;
- новую эпоху, которая не продолжает сохранённую историю;
- изменение корня для той же эпохи без создания доказательства раздвоения;
- пропуск обязательной проверки неизменности, когда она нужна для боевого
  решения.

Для локального этапа нужно реализовать проверку состояния и тесты. Для полного
боевого запуска сервер позже должен отдавать доказательства неизменности дерева
по образцу Certificate Transparency и Key Transparency.

### 4. Память свидетеля

Добавить локальную модель состояния свидетеля. Свидетель для пары
`журнал + эпоха` может подписать только один корень и один размер журнала. Если
ему предлагают второй другой корень для той же эпохи, он должен вернуть отказ
или доказательство собственной попытки раздвоения.

Это не заменяет независимых настоящих свидетелей, но закрывает ошибку модели:
свидетель не должен быть беспамятной функцией подписи.

### 5. Решение доверия

Добавить явный результат проверки:

- принято: подписи, цепочка и локальная история согласованы;
- нужно наблюдение: одиночная голова валидна, но ещё не подтверждена обменом;
- обнаружено раздвоение: есть две валидные разные головы одной эпохи;
- отказ: подписи, эпоха, размер, время или цепочка невалидны.

Боевые операции с высоким риском, особенно новое устройство, ротация identity и
восстановление, не должны считать состояние окончательно безопасным, если
результат “нужно наблюдение”.

## Что не входит в этот этап

- Не трогаем `rust_1mlrd`.
- Не пишем настоящую серверную интеграцию.
- Не проверяем реальные Android/iOS устройства.
- Не заявляем, что живая сеть уже доказала отсутствие раздвоения.
- Не публикуем личные данные пользователей.

## Тесты

Нужны реальные атакующие тесты:

- три злых свидетеля подписывают две разные головы одной эпохи, обе головы
  локально проходят порог;
- сравнение двух наблюдений создаёт доказательство раздвоения;
- конфликт без достаточных подписей не считается доказательством атаки, а
  отклоняется как мусор;
- повтор старой головы не двигает локальное состояние;
- новая эпоха без связи с предыдущей отклоняется в строгом режиме;
- свидетель отказывается подписывать второй другой корень для той же эпохи;
- доказательство раздвоения можно сериализовать и проверить обратно без потери
  смысла;
- публичное наблюдение не содержит account_id, телефон, имя, список устройств,
  контакты или чаты.

Формулировка должна быть честной: свидетели повышают цену атаки, а наблюдения и
проверка цепочки делают раздвоение доказуемым. Одиночная локальная проверка не
может доказать отсутствие другой версии, если порог свидетелей злой.

## Критерий готовности

Итерация считается закрытой только если:

- спецификация закоммичена;
- есть план внедрения;
- код добавляет библиотечные типы и проверки, а не только тестовый helper;
- атакующие тесты сначала воспроизводят дыру, затем проходят после исправления;
- все изменённые документы говорят одну и ту же правду;
- `cargo test -p umbrella-kt --all-features --locked` проходит;
- общий выпускной аудит обновлён или честно указывает оставшуюся внешнюю
  границу.
