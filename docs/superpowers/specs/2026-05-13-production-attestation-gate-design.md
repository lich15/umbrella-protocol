# Дизайн: серверная проверка устройства

Дата: 2026-05-13

## Цель

Сделать единый строгий слой серверной проверки устройства для двух боевых
запросов:

- развёртка облачного ключа в `umbrella-backup`;
- поиск контактов через `umbrella-oprf`.

Сервер не должен принимать запрос только потому, что у него правильная подпись
устройства или токен похож на настоящий. Запрос можно принять только если
одновременно прошли:

1. форма запроса;
2. подпись устройства;
3. свежий серверный вызов;
4. состояние устройства в журнале ключей;
5. запрет тестовой платформы;
6. настоящий платформенный проверяющий.

Если настоящий проверяющий для iOS, Android или Web ещё не подключён, боевой
путь закрыто отказывает понятной ошибкой.

## Текущая правда

Сейчас подпись запроса уже проверяется.

`Platform::Testing` уже отвергается в боевой проверке для `umbrella-backup` и
`umbrella-oprf`.

iOS, Android и Web пока не принимаются в бою: код честно возвращает ошибку о
неподключённом боевом проверяющем. Это безопасно, но слой проверки ещё слишком
тонкий: он не описывает общий серверный порядок проверки, не связывает
свежесть, состояние устройства и платформенную проверку в один обязательный
контракт.

Документы также должны быть выровнены: старые планы Фазы 1 и Фазы 2 содержат
незаполненные чекбоксы и старые фрагменты текста. Они полезны как история
планирования, но не должны выглядеть как текущая правда после уже слитых
исправлений.

## Выбранный подход

Выбран строгий серверный каркас проверки устройства.

Мы не начинаем с полной реализации Apple App Attest, Android Play Integrity или
WebAuthn. Сначала строим общий проверяющий слой, который невозможно обойти
тестовым токеном или неполным путём.

Платформенные проверки подключаются через отдельный интерфейс. Пока настоящая
реализация не передана, интерфейс возвращает отказ, а не успех.

Такой порядок сохраняет честность:

- серверная политика становится явной;
- тестовая платформа не может попасть в бой;
- будущие iOS, Android и Web проверки подключаются в одно заранее проверенное
  место;
- публичный запуск клиента остаётся закрыт до полной связки.

## Архитектура

Появляется общий набор серверных типов проверки, который можно использовать из
`umbrella-backup` и `umbrella-oprf`.

Логика делится на четыре части.

### 1. Контекст проверки

Контекст содержит всё, что сервер знает до выдачи ответа:

- текущее серверное время;
- допустимое окно свежести;
- серверный вызов, который должен совпасть с запросом;
- сведения об устройстве из журнала ключей;
- платформенный проверяющий;
- режим проверки: тестовый или боевой.

Боевой режим нельзя создать без явного платформенного проверяющего. Если
проверяющий отсутствует, создание боевого контекста или сама проверка должны
возвращать явный отказ.

### 2. Состояние устройства

Состояние устройства должно быть отдельным входом, а не скрытой догадкой:

- активное устройство принимается;
- неизвестное устройство отвергается;
- ожидающее устройство отвергается;
- отозванное устройство отвергается;
- устройство с будущей датой разрешения отвергается;
- устройство, заблокированное границей истории, отвергается.

Этот слой не должен откатываться к старому списку разрешённых ключей, если
журнал ключей уже содержит записи авторизации.

### 3. Платформенный проверяющий

Платформенный проверяющий получает:

- платформу;
- байты токена;
- серверный вызов;
- публичный ключ устройства;
- идентификатор приложения или сайта;
- текущее время.

Он возвращает либо подтверждение, либо точную причину отказа.

Первый вариант реализации в этой фазе:

- `Platform::Testing` всегда отвергается в боевом режиме;
- iOS, Android и Web возвращают "проверяющий не подключён", если нет настоящей
  реализации;
- тестовые проверки могут использовать отдельный тестовый проверяющий, но его
  нельзя передать как боевой без явного типа или имени.

### 4. Общий порядок проверки

Боевой серверный путь обязан идти в одном порядке:

1. разобрать запрос;
2. проверить размеры и форму;
3. проверить подпись устройства;
4. проверить совпадение серверного вызова;
5. проверить свежесть времени;
6. проверить состояние устройства;
7. проверить платформенный токен;
8. только после этого выдавать долю ключа или OPRF-ответ.

Если ранний шаг отказал, поздние шаги не вызываются. Например, плохая подпись
не должна доходить до платформенного проверяющего.

## Ошибки

Ошибки должны быть раздельными и проверяемыми:

- плохая подпись;
- неизвестное устройство;
- устройство отозвано;
- устройство ещё не активно;
- серверный вызов не совпал;
- серверный вызов устарел;
- время запроса в будущем;
- тестовая платформа в боевом режиме;
- настоящий платформенный проверяющий не подключён;
- токен пустой, слишком большой или не той формы.

Для внешних ошибок можно сохранять общий публичный текст, но внутри тестов
должна быть возможность проверить точную причину.

## Реальные атачные тесты

Тесты должны доказывать защиту, а не совпадение с удобной заглушкой.

Нужны проверки:

1. `Platform::Testing` с правильной подписью отвергается в боевом режиме.
2. iOS, Android и Web с правильной подписью не принимаются без настоящего
   платформенного проверяющего.
3. Плохая подпись отвергается до вызова платформенного проверяющего.
4. Чужой серверный вызов отвергается.
5. Старый серверный вызов отвергается.
6. Время из будущего отвергается.
7. Неизвестное устройство отвергается.
8. Отозванное устройство отвергается.
9. Ожидающее устройство отвергается.
10. Активное устройство доходит до платформенной проверки.
11. Пустой проверяющий, который всегда возвращает успех, нельзя использовать
    как боевой путь без явного тестового имени.
12. `umbrella-backup` и `umbrella-oprf` используют одинаковую политику для
    платформ и свежести.

## Что меняем

План внедрения должен затронуть:

- `crates/umbrella-backup/src/cloud_wrap/signed_request.rs`;
- `crates/umbrella-oprf/src/attestation.rs`;
- ошибки `umbrella-backup` и `umbrella-oprf`;
- при необходимости новый общий модуль или малый общий крейт, если дублирование
  правил начнёт мешать;
- тесты для обеих подсистем;
- `docs/security/production-readiness-boundaries.md`;
- документы планов, где старые чекбоксы или старый текст могут выглядеть как
  текущая правда.

## Что не меняем

В этой фазе не делаем:

- полный Apple App Attest;
- полный Android Play Integrity;
- полный WebAuthn;
- открытие публичного запуска клиента;
- настоящую серверную инфраструктуру;
- мобильную связку через внешний интерфейс.

Эти части идут отдельными фазами после того, как общий серверный слой станет
строгим и проверенным.

## Проверки перед завершением

Минимальные ворота:

```bash
cargo test -p umbrella-backup --all-features --locked
cargo test -p umbrella-oprf --all-features --locked
cargo test -p umbrella-client --all-features --locked
cargo test -p umbrella-ffi public_bootstrap --locked
cargo test -p umbrella-ffi public_bootstrap --features pq --locked
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --workspace --all-features --locked
cargo test --workspace --all-features --locked
```

Если полный набор слишком долгий для промежуточной итерации, точечные проверки
обязательны перед каждым коммитом, а полный набор обязателен перед завершением
ветки.

## Свежие источники

Проверено 2026-05-13:

- Apple App Attest: https://developer.apple.com/documentation/devicecheck/validating-apps-that-connect-to-your-server
- Android Play Integrity: https://developer.android.com/google/play/integrity
- WebAuthn Level 3: https://www.w3.org/TR/webauthn/all/

Из этих источников для текущей фазы важно одно правило: сервер должен проверять
свежий вызов, подпись, привязку к приложению или сайту и результат платформы.
Нельзя принимать токен только потому, что он непустой.

## Готовность фазы

Фаза готова, когда:

- боевой серверный слой имеет один явный порядок проверки;
- тестовая платформа не проходит ни для развёртки ключа, ни для OPRF;
- настоящие платформы без настоящего проверяющего закрыто отказывают;
- активное устройство доходит до платформенной проверки, а неизвестное,
  ожидающее и отозванное устройство — нет;
- документы не обещают готовую iOS, Android или Web проверку;
- публичный запуск клиента остаётся закрыт по честной причине;
- рабочая папка чистая после коммита.
